# Dynamic Workers

On Cloudflare, Docsflare deploys one Worker for a Mintlify-style project. That is a straightforward shape for self-hosting: content is pre-rendered during `docsflare build`, bundled into the Fetch handler, and can be searched through one Cloudflare AI Search instance.

Cloudflare Dynamic Workers are a good next step for hosted, multi-site docs. A loader Worker can receive a site ID, read that site's generated Worker code and assets from storage, and run it through a `worker_loaders` binding.

```json
{
  "worker_loaders": [
    {
      "binding": "LOADER"
    }
  ]
}
```

The loader shape would be:

1. Store each imported Mintlify project as generated content, assets, and search metadata.
2. Use `env.LOADER.get(siteVersionId, callback)` to cache a per-site Worker isolate.
3. Return `WorkerCode` from the callback with the compiled docs Worker module and controlled bindings.
4. Keep outbound network access disabled for tenant Workers unless a site explicitly needs it.
5. Serve static assets through a controlled binding backed by R2 or KV.

Dynamic Workers are intentionally not required for the current single-site implementation or for non-Cloudflare targets. The Cloudflare path stays simple with normal Wrangler commands, while the same renderer can run through the Node.js or Fetch API integration paths.
