# Documentation without the platform lock-in

Docsflare turns a Mintlify-style content directory into a portable Fetch API application. Keep familiar `docs.json`, MDX components, navigation, search, and theming—then run the same generated site on Cloudflare Workers, Node.js, or another Fetch-compatible runtime.

<Info>
Cloudflare is the preferred deployment target, not a dependency. Local search, static assets, sitemap, `llms.txt`, and every documentation page work without a Cloudflare account.
</Info>

## One content tree, several runtimes

The build output has no hosted control plane. Pick the runtime that fits each environment without rewriting the docs.

<Tabs>
  <Tab title="Cloudflare Workers">
    ```bash
    docsflare deploy docs
    ```

    Deploy the generated Worker with optional Cloudflare AI Search and observability configuration.
  </Tab>
  <Tab title="Node.js">
    ```bash
    docsflare build docs
    docsflare start docs
    ```

    Run the built-in Node adapter on any host that supports Node.js 20 or newer.
  </Tab>
  <Tab title="Fetch handler">
    ```typescript
    import { handleRequest } from "./docs/.docsflare/worker";

    const response = await handleRequest(new Request(url), env, context);
    ```

    Adapt the generated standard request handler to another serverless or edge runtime.
  </Tab>
</Tabs>

## From repository to production

<Steps>
  <Step title="Point Docsflare at your content">
    Start fresh with `docsflare init docs` or use an existing directory containing `docs.json` or `mint.json`.
  </Step>
  <Step title="Preview the complete application">
    Run `docsflare dev docs` and test navigation, responsive layouts, color modes, search, and MDX interactions locally.
  </Step>
  <Step title="Build once">
    Docsflare renders pages and assets into `.docsflare/`, including the portable Worker and Node adapter.
  </Step>
  <Step title="Ship to your preferred runtime">
    Deploy directly to Cloudflare or package the generated Fetch handler for your platform.
  </Step>
</Steps>

<Tip>
The default local index keeps search useful everywhere. Add one `AI_SEARCH` binding when you want Cloudflare's managed crawler, hybrid retrieval, and generated answers in production.
</Tip>

## A small configuration surface

The common path stays intentionally short:

<CodeGroup>
```json docs.json
{
  "name": "Acme Docs",
  "colors": {
    "primary": "#0f766e",
    "light": "#2dd4bf",
    "dark": "#134e4a"
  },
  "navigation": [
    {
      "group": "Guides",
      "pages": ["introduction", "quickstart"]
    }
  ]
}
```

```css custom.css
:root {
  --docsflare-content-width: 760px;
  --docsflare-radius: 14px;
}

[data-docsflare-component="card"] {
  box-shadow: 0 16px 40px rgb(15 23 42 / 8%);
}
```
</CodeGroup>

| Capability | Portable default | Cloudflare enhancement |
| --- | --- | --- |
| Rendering | Build-time MDX and static assets | Globally distributed Worker |
| Search | In-memory page index | Managed AI Search crawler |
| Chat | Relevant source-page responses | Retrieval-grounded model response |
| Runtime | Standard Fetch handler and Node adapter | Wrangler deploy workflow |
| Styling | Design tokens and project CSS | Same appearance, no vendor fork |

<Accordion title="Can I migrate an existing Mintlify project?">
Usually. Keep the content and navigation, then replace unsupported custom React imports. Common documentation components and appearance fields have built-in equivalents.
</Accordion>

<Accordion title="Does custom CSS require a paid plan?">
No. Every `.css` file below the content directory is discovered and bundled automatically. Project styles load after the built-in styles.
</Accordion>

<Accordion title="Am I required to use Cloudflare AI Search?">
No. Search and source-based chat have portable local behavior. Cloudflare AI Search is an optional production upgrade through the `AI_SEARCH` binding.
</Accordion>
